## Краткое описание работы

В ходе данной лабораторной работы были изучены принципы архитектуры REST и создание серверной части приложения на базе ASP.NET Core Web API. Основной упор был сделан на работу с контроллерами, настройку маршрутизации, использование DTO для защиты данных и тестирование API с помощью Swagger UI.

## Список реализованных маршрутов

| Метод      | Маршрут           | Описание                                 | Параметры                        | Ответ (успех)    |
|------------|-------------------|------------------------------------------|----------------------------------|------------------|
| **GET**    | `/api/tasks`      | Получение списка всех задач              | `completed` (query, опционально) | 200 OK           |
| **GET**    | `/api/tasks/{id}` | Получение задачи по идентификатору       | `id` (path)                      | 200 OK / 404     |
| **POST**   | `/api/tasks`      | Создание новой задачи                    | `CreateTaskDto` (body)           | 201 Created      |
| **PUT**    | `/api/tasks/{id}` | Полное обновление задачи                 | `id` (path), `TaskItem` (body)   | 204 No Content   |
| **DELETE** | `/api/tasks/{id}` | Удаление задачи из списка                | `id` (path)                      | 204 No Content   |

|Маршруты              | [HttpGet] атрибут над методом          |

## Главные выводы

1.	**REST** — не протокол, а архитектурный стиль. Те же принципы работают с любым языком и фреймворком.
2.	**Контроллер** в ASP.NET Core = Router в Express, только с автоматической документацией и строгой типизацией.
3.	**DTO защищает API от некорректных данных**: клиент передаёт только то, что сервер разрешает принять.
4.	**Swagger UI** позволяет тестировать API без Postman и без написания тестового JavaScript-кода.
5.	**Правильные HTTP-статусы** — часть «контракта» API. Клиент должен понимать, что произошло, по коду ответа.
